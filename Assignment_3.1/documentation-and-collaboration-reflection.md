# Documentation and Collaboration Reflection

## What Assumptions the Original README Made About Its Audience

The original README assumed that anyone reading it already knew what the Conference Room Booking System was, who built it, and how to set it up. There were no setup instructions, no explanation of the folder structure, and no guidance on how to contribute. It was written for someone already on the team rather than someone joining it for the first time. A new developer landing on the repository would have had to ask Nido, Thapelo, Caitlin, or Thato multiple questions just to get the project running locally. That kind of knowledge gap creates dependency on specific people and slows the whole team down.

## How the Updated README Improves Onboarding

The updated README removes the need for a new developer to ask anyone for help on their first day. It explains what the system does and who it serves, how to set it up step by step from cloning the repo to starting the server, what every folder in the project contains, where to find the Sprint 1 documentation, and exactly how to raise a Pull Request and write a commit message. The planned sections area also signals honestly what documentation is still missing so new team members know what is complete and what is not yet there.

## What I Learned About Communication Through Pull Requests and Issues

Writing a Pull Request description forced me to explain my reasoning not just my changes. A reviewer reading the PR description should understand why something was done before they even open a single file. Issues work the same way. A well structured bug report means the next person who picks it up understands the problem fully without needing to track down the person who raised it. Both Pull Requests and issues replace informal conversations that would otherwise happen in messages or meetings and they create a permanent record that the whole team can refer back to.
